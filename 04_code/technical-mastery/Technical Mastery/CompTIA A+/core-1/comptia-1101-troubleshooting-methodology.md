# comptia-troubleshooting-methodology

## 🔥 Overview
The CompTIA 6-Step Troubleshooting Methodology provides a structured, disciplined framework for diagnosing and resolving technical issues efficiently. Mastering this process is crucial for IT support roles and is a core component of the A+ certification, ensuring problems are addressed systematically rather than haphazardly.

## 📌 The 6 Steps (Key Concepts)

1.  **Identify the Problem:** (Gather Information & Scope the Issue)
    *   **Gather information:** Question the user (open-ended: who, what, when, where, why?), identify user changes (software installs, hardware changes, updates?).
    *   **Inquire about environmental or infrastructure changes.**
    *   **Review system and application logs** (Event Viewer, console logs). Note specific error messages.
    *   **Backup data:** Before making changes, ensure critical data is backed up if possible and appropriate.
    *   **Duplicate the problem** if possible, or clearly identify the exact symptoms.
    *   *Key Idea: Understand precisely what is wrong and what might have changed.*

2.  **Establish a Theory of Probable Cause:** (Hypothesize - Start Simple)
    *   **Question the obvious:** Don't overlook simple fixes (power, connections, reboot, correct input?).
    *   **Formulate initial hypotheses:** Based on symptoms, list the most likely causes first (e.g., update issue, user error, hardware failure, network config, malware).
    *   **Consider multiple approaches/layers:** Physical, OS, Application, Network, User.
    *   **Research (if needed):** Look up error codes, symptoms, known issues.
    *   *Key Idea: Make an educated guess about the root cause, prioritizing simplicity.*

3.  **Test the Theory to Determine Cause:** (Isolate the Variable)
    *   **Test ONE theory at a time.** Change only one variable and observe the result.
    *   **If theory is confirmed:** Proceed to Step 4.
    *   **If theory is NOT confirmed:**
        *   **Establish a *new* theory** based on test results (return to Step 2).
        *   **Undo the change** made during the test before trying the next theory.
    *   **Escalate (if necessary):** If unable to determine cause, consult senior techs, vendors, or documentation.
    *   *Key Idea: Prove or disprove your hypothesis methodically and cleanly.*

4.  **Establish a Plan of Action to Resolve the Problem and Implement the Solution:** (Design & Execute the Fix)
    *   Outline specific steps needed based on the confirmed cause.
    *   Consider potential side effects or impacts (downtime, data risk, config changes).
    *   Have a rollback plan if feasible.
    *   **Implement the plan:** Execute the steps carefully.
    *   *Key Idea: Strategize the repair and then perform it.*

5.  **Verify Full System Functionality and, If Applicable, Implement Preventive Measures:** (Confirm & Prevent Recurrence)
    *   **Confirm the fix:** Ensure the original reported problem is resolved.
    *   **Test related functionality:** Check that the fix didn't introduce new problems. Involve the user in testing if possible.
    *   **Implement preventive measures:** What can prevent recurrence? (e.g., Updates, user education, monitoring, configuration changes).
    *   *Key Idea: Ensure the solution worked completely and fortify against future issues.*

6.  **Document Findings, Actions, and Outcomes:** (Record for the Archives)
    *   **Record:** Initial problem/symptoms, theories tested, confirmed cause, resolution steps, final outcome, parts/settings involved.
    *   **Purpose:** Build knowledge base, track recurring issues, provide accountability, analyze trends.
    *   *Key Idea: Leave a clear, concise record for future reference and analysis.*
